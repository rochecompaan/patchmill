# Blocked triage design

## Purpose

Patchmill triage needs to distinguish issues that are suitable for agents but
cannot start yet because other same-repository issues must be completed first.
Today triage only has `agent-ready`, `needs-info`, and `agent-unsuitable`, so
dependency-blocked work can be mislabeled as ready or treated as missing
information.

This design adds an explicit `blocked` triage bucket, applies a clear repository
label and comment, and lets later triage runs move blocked issues to
`agent-ready` once all referenced blockers are closed.

## Goals

- Add `blocked` as a canonical triage bucket and default repository label.
- Require concrete same-repository issue numbers for blocked classification.
- Record structured blocker metadata as `blockedBy: number[]`.
- Comment clearly on blocked issues with `Blocked by: #1, #2` style wording.
- On later triage runs, remove `blocked` and add `agent-ready` when all blockers
  are closed.
- Leave old blocked comments intact and post a new unblock comment.

## Non-goals

- Cross-repository dependency tracking.
- Free-text blocker tracking without issue numbers.
- Editing older Patchmill comments during unblock.
- Treating the configured `done` label as equivalent to closed for unblock
  decisions.

## Triage state model

Patchmill will support four canonical triage buckets:

- `agent-ready`
- `needs-info`
- `agent-unsuitable`
- `blocked`

Default config will include a blocked label, for example:

```json
{
  "labels": {
    "ready": "agent-ready",
    "needsInfo": "needs-info",
    "unsuitable": "agent-unsuitable",
    "blocked": "blocked"
  },
  "triage": {
    "stateMap": {
      "agent-ready": "agent-ready",
      "needs-info": "needs-info",
      "agent-unsuitable": "agent-unsuitable",
      "blocked": "blocked"
    }
  }
}
```

`triage.stateMap` validation must accept `blocked`. Config normalization must
make the blocked label part of the triage-state label set so label replacement
removes stale triage-state labels before applying the selected bucket.

## Triage output contract

Triage preview/result entries will include structured blocker metadata:

```json
{
  "issueNumber": 3,
  "currentLabels": [],
  "proposedLabels": ["blocked"],
  "blockedBy": [1, 2],
  "rationale": "The create-poll form depends on the app scaffold and poll data model.",
  "wouldComment": "...",
  "wouldClose": false,
  "questions": []
}
```

Rules:

- `blockedBy` is required for `blocked` entries and must contain at least one
  positive same-repository issue number.
- `blockedBy` should be empty or omitted for non-blocked entries.
- If the triage agent can identify dependency blockers but cannot name concrete
  issue numbers, it must classify the issue as `needs-info` instead.
- If a blocked result has missing, invalid, duplicate-only, self-referential, or
  unknown blocker numbers, Patchmill should reject the result or coerce it to
  `needs-info` before mutation. It must not apply a misleading `blocked` label.

## Prompt and skill behavior

The bundled triage skill and dry-run prompt will describe `blocked` as work that
is clear and agent-suitable but cannot start until specific same-repository
issues are closed.

The prompt should instruct the triage agent to:

- infer dependencies from issue titles, bodies, comments, labels, and
  chronological context;
- use `blocked` only when concrete issue numbers can be listed;
- include those numbers in `blockedBy`;
- include `Blocked by: #N` or `Blocked by: #N, #M` in the proposed comment;
- use `needs-info` when the dependency exists but cannot be tied to issue
  numbers.

Issue content remains untrusted input. Maintainer/system prompt instructions and
the configured triage skill stay authoritative.

## Execute behavior

When triage applies a blocked classification:

1. Remove other triage-state labels from the issue.
2. Add the configured blocked label.
3. Post a Patchmill triage comment using the existing AI-generated prefix and
   clear blocker wording, for example:

   ```markdown
   > _This was generated by AI during triage._

   Blocked by: #1, #2

   This issue is clear and suitable for automation, but it should wait until the
   app scaffold and domain model are complete.
   ```

4. Record `primaryBucket: "blocked"`, `blockedBy`, previous/final labels,
   rationale, and comment in the triage log.

Patchmill should avoid duplicate noisy blocked comments on later runs if the
issue remains blocked by the same set of open issues. If the blocker list
changes, a fresh comment is acceptable because it communicates new state.

## Re-triage and auto-unblock

On later triage runs, Patchmill re-evaluates issues currently labeled `blocked`.

For each blocked issue, Patchmill should recover the structured blocker list
from the current triage result and/or previous Patchmill triage metadata. It
then checks those same-repository issues through the configured host provider.

If all referenced blocker issues are closed:

1. Remove the configured blocked label.
2. Add the configured ready label.
3. Post a new Patchmill triage comment, leaving old blocked comments intact, for
   example:

   ```markdown
   > _This was generated by AI during triage._

   The blocking issues are now closed: #1, #2.

   This issue is now agent-ready.
   ```

4. Record the state transition in the triage log with
   `previousState: "blocked"`, `finalState: "agent-ready"`, and `blockedBy`.

If any referenced blocker remains open:

- keep the configured blocked label;
- do not add `agent-ready`;
- record the still-open blockers in the log;
- avoid posting another comment unless the blocker list changes.

If the blocker list cannot be recovered, Patchmill should re-run normal triage
and prefer `needs-info` over assuming the issue is unblocked.

The only automatic unblock rule is that all referenced blockers are closed. The
configured `done` label is not enough by itself.

## Host provider expectations

Auto-unblock requires the host layer to provide enough issue state to determine
whether blocker issue numbers are closed. Providers should expose or reuse issue
lookup/list behavior that returns at least:

- issue number;
- state/open-closed status;
- labels;
- title, body, and comments for normal triage context.

If a provider cannot resolve a referenced blocker, Patchmill should treat the
blocked classification as invalid for mutation and prefer `needs-info` or a
failed preview entry rather than silently unblocking.

## Logging and reporting

Triage logs and progress output should surface blocked state clearly:

- `primaryBucket: "blocked"`
- `blockedBy: number[]`
- previous/final labels
- whether a blocked issue stayed blocked or became `agent-ready`
- mutation status and any provider lookup errors

Dry-run output should show the labels and comments that would be applied without
mutating repository state.

## Testing plan

Automated tests should cover reusable and regression-prone behavior:

- config/default/state-map validation accepts `blocked` and rejects unknown
  buckets;
- label policy treats `blocked` as a triage-state label that replaces other
  triage-state labels;
- dry-run prompt includes the `blockedBy` contract and blocked classification
  rules;
- parser validation accepts valid blocked results and rejects/coerces invalid
  `blockedBy` results;
- execute mode applies `blocked`, removes stale triage labels, posts the blocked
  comment, and logs `blockedBy`;
- re-triage keeps an issue blocked while any blocker is open;
- re-triage removes `blocked`, adds `agent-ready`, and posts an unblock comment
  when all blockers are closed;
- fixture test repo triage expectations include dependency-blocked issues.

Documentation-only changes should be verified by markdown review or existing
documentation checks rather than new tests.

## Documentation updates

Update configuration and workflow docs to explain:

- `blocked` as a canonical triage bucket;
- default blocked label configuration;
- `blockedBy` issue-number requirements;
- blocked comments and unblock comments;
- auto-unblock when all referenced blockers are closed;
- the distinction between `blocked` and `needs-info`.
