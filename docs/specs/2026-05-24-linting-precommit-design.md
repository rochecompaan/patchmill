# Linting and Pre-commit Design

## Goal

Add repository linting for TypeScript and Markdown, repository formatting with Prettier, and a Node-native pre-commit hook that auto-formats staged files before running lint checks.

## Current Context

Patchmill is a private Node project with TypeScript source and test files under `bin/`, `scripts/`, `src/`, and `test-support/`. The repository already has npm test scripts but no existing ESLint, Markdown lint, Husky, or lint-staged configuration. There are existing uncommitted documentation changes in the working tree; the linting work should avoid overwriting unrelated edits.

## Chosen Approach

Use Prettier for formatting, ESLint flat config with `typescript-eslint` for TypeScript linting, `markdownlint-cli2` for Markdown linting, and Husky plus lint-staged for pre-commit enforcement.

This is the best fit because it keeps all tooling Node-native, avoids introducing Python `pre-commit`, makes commit-time checks fast by operating on staged files, and gives Markdown prose consistent 80-character wrapping through Prettier.

## Alternatives Considered

1. Add only npm lint scripts. This is simpler but relies on contributors remembering to run them before commits.
2. Add Husky and lint-staged with TypeScript and Markdown linters. This adds light commit enforcement while keeping the workflow fast, but it does not provide consistent Markdown wrapping.
3. Add Prettier with Husky and lint-staged. This adds formatting enforcement and is preferred because Markdown prose should be wrapped at 80 characters automatically.

## Files and Responsibilities

- `eslint.config.js`: ESLint flat configuration for TypeScript files in the repo.
- `.markdownlint-cli2.jsonc`: Markdown lint configuration with practical defaults for project docs.
- `.prettierrc.json`: Prettier formatting configuration with 80-character Markdown prose wrapping.
- `.prettierignore`: Ignore generated, dependency, and local state paths.
- `.husky/pre-commit`: Git hook that runs lint-staged.
- `package.json`: Adds lint and format scripts, a Husky prepare script, development dependencies, and lint-staged file patterns.
- `package-lock.json`: Captures the exact npm dependency graph if generated by `npm install`.

## TypeScript Linting Design

ESLint should lint `*.ts` files under source, script, bin, and test-support paths. The configuration should use `@typescript-eslint/parser` and the recommended TypeScript rules through the flat-config-friendly `typescript-eslint` package. The lint command should fail on warnings so CI or local usage can treat linting as a clear pass/fail gate.

The configuration should account for this repo's Node runtime and current ESM style. If type-aware linting requires more setup than the current repo has, start with syntax-aware recommended rules and keep the configuration small.

## Markdown Linting Design

Markdown linting should cover `README.md` and files under `docs/`. The configuration should use `markdownlint-cli2` because it is Node-native and easy to call from npm scripts and lint-staged.

Rules should be practical for this documentation-heavy repo. Markdown line-length linting should not enforce wrapping because Prettier will handle prose wrapping, while URLs, tables, headings, and code blocks may legitimately exceed 80 characters. Generated or local state directories should be ignored if needed.

## Formatting Design

Prettier should format TypeScript, JavaScript, JSON, and Markdown files. The configuration should set `printWidth` to `80` and `proseWrap` to `always`, which wraps Markdown prose around 80 characters without forcing long URLs or code blocks to wrap.

Formatting scripts should include a check mode for verification and a write mode for local cleanup. Pre-commit should use write mode on staged files so commits auto-format rather than failing only because formatting has not been applied yet.

## Pre-commit Design

Use Husky for hook installation and lint-staged for staged-file selection. The `prepare` script should install Husky hooks when dependencies are installed. The pre-commit hook should run `npx lint-staged`.

lint-staged should run Prettier first on staged supported files, then ESLint on staged TypeScript files and markdownlint-cli2 on staged Markdown files. It should not lint the entire repo on every commit, keeping commits fast.

## Verification

Verification should include:

1. Running the TypeScript lint script.
2. Running the Markdown lint script.
3. Running the Prettier check script.
4. Running the combined lint script.
5. Running the existing test suite to confirm no unrelated breakage.
6. If possible, running lint-staged in a dry or direct mode to confirm the hook command resolves.

## Out of Scope

- Adding CI workflow changes.
- Refactoring existing TypeScript code beyond changes needed to satisfy the newly introduced lint rules.
- Rewriting existing documentation except for minimal lint fixes required by the Markdown linter.
