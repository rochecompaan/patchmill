# Linting and Pre-commit Design

## Goal

Add repository linting for TypeScript and Markdown, with a Node-native pre-commit hook that runs linting on staged files before commits.

## Current Context

Patchmill is a private Node project with TypeScript source and test files under `bin/`, `scripts/`, `src/`, and `test-support/`. The repository already has npm test scripts but no existing ESLint, Markdown lint, Husky, or lint-staged configuration. There are existing uncommitted documentation changes in the working tree; the linting work should avoid overwriting unrelated edits.

## Chosen Approach

Use ESLint flat config with `typescript-eslint` for TypeScript, `markdownlint-cli2` for Markdown, and Husky plus lint-staged for pre-commit enforcement.

This is the best fit because it keeps all tooling Node-native, avoids introducing Python `pre-commit`, and makes commit-time checks fast by linting only staged files.

## Alternatives Considered

1. Add only npm lint scripts. This is simpler but relies on contributors remembering to run them before commits.
2. Add Husky and lint-staged with TypeScript and Markdown linters. This adds light commit enforcement while keeping the workflow fast.
3. Add formatting tooling such as Prettier at the same time. This is useful later, but it expands scope beyond the requested linting setup.

## Files and Responsibilities

- `eslint.config.js`: ESLint flat configuration for TypeScript files in the repo.
- `.markdownlint-cli2.jsonc`: Markdown lint configuration with practical defaults for project docs.
- `.husky/pre-commit`: Git hook that runs lint-staged.
- `package.json`: Adds lint scripts, a Husky prepare script, development dependencies, and lint-staged file patterns.
- `package-lock.json`: Captures the exact npm dependency graph if generated by `npm install`.

## TypeScript Linting Design

ESLint should lint `*.ts` files under source, script, bin, and test-support paths. The configuration should use `@typescript-eslint/parser` and the recommended TypeScript rules through the flat-config-friendly `typescript-eslint` package. The lint command should fail on warnings so CI or local usage can treat linting as a clear pass/fail gate.

The configuration should account for this repo's Node runtime and current ESM style. If type-aware linting requires more setup than the current repo has, start with syntax-aware recommended rules and keep the configuration small.

## Markdown Linting Design

Markdown linting should cover `README.md` and files under `docs/`. The configuration should use `markdownlint-cli2` because it is Node-native and easy to call from npm scripts and lint-staged.

Rules should be practical for this documentation-heavy repo. Long lines should not fail linting because docs can contain URLs, commands, and prose that are easier to maintain without aggressive wrapping. Generated or local state directories should be ignored if needed.

## Pre-commit Design

Use Husky for hook installation and lint-staged for staged-file selection. The `prepare` script should install Husky hooks when dependencies are installed. The pre-commit hook should run `npx lint-staged`.

lint-staged should run ESLint only on staged TypeScript files and markdownlint-cli2 only on staged Markdown files. It should not lint the entire repo on every commit, keeping commits fast.

## Verification

Verification should include:

1. Running the TypeScript lint script.
2. Running the Markdown lint script.
3. Running the combined lint script.
4. Running the existing test suite to confirm no unrelated breakage.
5. If possible, running lint-staged in a dry or direct mode to confirm the hook command resolves.

## Out of Scope

- Adding Prettier or automatic formatting.
- Adding CI workflow changes.
- Refactoring existing TypeScript code beyond changes needed to satisfy the newly introduced lint rules.
- Rewriting existing documentation except for minimal lint fixes required by the Markdown linter.
