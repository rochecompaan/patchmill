# Patchmill.dev Docs Site Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use
> superpowers:subagent-driven-development (recommended) or
> superpowers:executing-plans to implement this plan task-by-task. Steps use
> checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build a static `patchmill.dev` documentation site in `site/` using
Astro and Starlight, with a polished landing page, public docs content, and
GitHub Pages deployment.

**Architecture:** Keep the Patchmill CLI/package at the repository root and add
a separate Astro app under `site/`. The landing page is a custom Astro page;
docs pages use Starlight content collections under `site/src/content/docs/`.
Root scripts delegate to the site package so maintainers can build and preview
the site from the repository root.

**Tech Stack:** Node.js 24-compatible npm project, Astro 7.0.6, Starlight
0.41.3, Markdown docs, GitHub Pages Actions deployment.

## Global Constraints

- Use `site/` as the dedicated docs-site directory.
- Keep the root package as the Patchmill package; do not move `bin/`, `src/`,
  `skills/`, `extensions/`, or `fixtures/`.
- Use Astro for the custom landing page.
- Use Starlight for documentation navigation, search, page layout, code
  highlighting, dark mode, and accessibility defaults.
- Deploy static output from `site/dist/` to GitHub Pages.
- Preserve Patchmill runtime behavior; this feature only adds docs-site files,
  docs content, scripts, and deployment configuration.
- Do not publish internal `docs/plans/` or `docs/specs/` content in the site.
- Do not add docs versioning, i18n, blog infrastructure, or plugin-heavy
  community features in this implementation.
- Use npm, not pnpm or Yarn.
- Root Node engine remains `>=22.19.0`; GitHub Actions should build with
  Node 24.
- Do not write automated tests merely to assert workflow YAML or static docs
  text; verify docs changes with build, lint, format, and existing checks.
- Because site npm dependencies will change, rerun the Nix package verification
  commands and commit `nix/package.nix` only if the npm dependency hash script
  updates it.

---
```markdown
title: What Patchmill is
description: Understand Patchmill's role as an agent-driven software factory.
---

Patchmill is an agent-driven software factory for turning product work into
reviewed, landed changes without hiding the engineering judgment between idea
and production.

It connects an issue host, repository policy, git worktrees, configurable
workflow skills, and Pi runtime instructions so a repository can move from open
product work to reviewed diffs with clear handoffs.

## The production-line model

Patchmill makes the stations in automated development explicit:

1. Intake product work from an issue host.
2. Sort what is ready to advance.
3. Write or reuse a plan.
4. Implement in an isolated worktree.
5. Review the result.
6. Collect evidence when the workflow asks for it.
7. Land the change.
8. Record what happened.

The goal is not a black box that writes code. The goal is a factory floor where
every station is visible, configurable, and designed to preserve software
craftsmanship while making iterative engineering scalable.

## Where Superpowers fits

Patchmill relies on the Superpowers skill pack for much of the workflow
discipline around planning, implementation, debugging, review, and verification.

In the factory metaphor, Patchmill provides the factory floor and Superpowers
provides much of the expertise that moves work through the factory.

## What to read next

- Start with the [quickstart](/getting-started/quickstart/).
- Learn how repository behavior is configured in
  [configuration](/getting-started/configuration/).
- Review [issue-agent workflows](/guides/issue-agent-workflows/) when you are
  ready to understand the full production line.
```

Run this script to update ignore and Markdown lint configuration:

```bash
python3 <<'PY'
from pathlib import Path

append_lines = {
    Path(".gitignore"): ["site/.astro/", "site/dist/"],
    Path(".prettierignore"): [
        "site/.astro/",
        "site/dist/",
        "site/package-lock.json",
        "site/**/*.astro",
    ],
}

for path, lines in append_lines.items():
    text = path.read_text()
    additions = [line for line in lines if line not in text.splitlines()]
    if additions:
        if not text.endswith("\n"):
            text += "\n"
        text += "\n" + "\n".join(additions) + "\n"
        path.write_text(text)

path = Path(".markdownlint-cli2.jsonc")
text = path.read_text()
old = '"globs": ["README.md", "docs/**/*.md", "skills/**/*.md"],'
new = '"globs": ["README.md", "docs/**/*.md", "skills/**/*.md", "site/src/content/docs/**/*.md"],'
if old not in text:
    raise SystemExit("Expected markdownlint globs line was not found")
text = text.replace(old, new)
if '"site/.astro/**"' not in text:
    text = text.replace(
        '"node_modules/**",\n',
        '"node_modules/**",\n    "site/.astro/**",\n    "site/dist/**",\n',
    )
path.write_text(text)
PY
```

```markdown
Patchmill is an agent-driven software factory for turning product work into

It connects an issue host, repository policy, git worktrees, configurable

## The production-line model

Patchmill makes the stations in automated development explicit:

1. Intake product work from an issue host.
2. Sort what is ready to advance.
3. Write or reuse a plan.
4. Implement in an isolated worktree.
5. Review the result.
6. Collect evidence when the workflow asks for it.
7. Land the change.
8. Record what happened.

The goal is not a black box that writes code. The goal is a factory floor where
every station is visible, configurable, and designed to preserve software
craftsmanship while making iterative engineering scalable.

Patchmill coordinates local repository automation:

- Host provider access for issue and pull-request workflows.
- Repository-local configuration in `patchmill.config.json`.
- Project-local skills under `.patchmill/skills/`.

discipline around planning, implementation, debugging, review, and verification.
```

````markdown
`patchmill run-once` advances one ready issue through the configured production
line. The workflow can reuse approved artifacts from an issue or create new
artifacts when the approval policy requires them.

The high-level sequence is:

1. Select a ready issue.
2. Check repository and provider preconditions.
3. Resolve or create the required spec and plan artifacts.
4. Claim the issue and create an isolated implementation worktree.
5. Run implementation with configured skills and runtime instructions.
6. Request review.
7. Collect evidence when the workflow asks for it.
8. Record run state and handoff information.

## Approval gates

Patchmill keeps gates explicit. A repository can require approved specs,
approved plans, implementation review, or evidence collection before work
advances.

Approval gates make automated work auditable: reviewers can see what was
planned, what changed, what was verified, and what remains blocked.

factory loop as development testing rather than supported usage.
````
